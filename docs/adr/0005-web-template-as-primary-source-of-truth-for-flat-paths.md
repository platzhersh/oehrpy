# ADR-0005: Web Template as Primary Source of Truth for FLAT Paths

**Date:** 2026-04-04

## Status

Accepted

## Context

oehrpy supports the openEHR FLAT format (simSDT JSON) for composing and validating
compositions against EHRBase. The FLAT format represents a composition as a flat
key-value map where keys are human-readable paths, for example:

```
vital_signs_observations/vital_signs/blood_pressure/systolic|magnitude
```

These paths are **not** directly derivable from the Operational Template (OPT) XML.
The OPT encodes constraints using archetype node IDs (`at0001`, `at0006`, etc.), while
FLAT paths are built from human-readable `id` fields that only exist in the **Web
Template** — the JSON representation of the template generated by the CDR at upload
time.

### The OPT Parsing Trap

oehrpy's initial approach used an `OPTParser` to extract archetype structure from OPT
XML and generate FLAT paths from it (see `BuilderGenerator` in
`src/openehr_sdk/templates/builder_generator.py`). The intent was to allow offline
builder generation without a live CDR.

This approach caused persistent, hard-to-diagnose failures. During integration testing
(documented in `docs/integration-testing-journey.md`), EHRBase consistently rejected
all FLAT compositions with `"Could not consume Parts"` errors, regardless of how the
paths were constructed from the OPT. The root cause:

- The OPT uses archetype node IDs (`at0006`) as node identifiers.
- The FLAT format uses template-level `id` fields (`any_event`, `pulse_heart_beat`,
  `vital_signs_observations`) that only exist in the Web Template tree.
- There is no reliable, general-purpose algorithm to derive the Web Template `id`
  values from OPT XML. The CDR (EHRBase) performs this transformation internally when
  the OPT is uploaded, applying normalisation rules (lowercasing, space-to-underscore
  conversion, deduplication suffixes) that are not fully documented and differ between
  CDR versions.

Concrete example of the mismatch discovered during integration testing:

| Source | Path segment |
|--------|-------------|
| OPT archetype ID | `openEHR-EHR-OBSERVATION.pulse.v1` |
| Naively derived (wrong) | `pulse` |
| Actual Web Template `id` | `pulse_heart_beat` |

Similarly, the composition root `id` (`vital_signs_observations`) cannot be derived
from the OPT concept name without knowing the exact normalisation the CDR applies.

### External Validation

This behaviour is confirmed by:

- The openEHR SDT specification, which states that FLAT paths are derived from the
  Simplified OPT (sOPT) / Web Template tree, not directly from the OPT.
- Ian McNicoll (openEHR community) on Discourse (2021): *"you do need to parse the Web
  template and figure out how the flat paths are constructed from the 'id' segments."*
- The EHRBase Java SDK (`openEHR_SDK`): its `FileBasedTemplateProvider` reads OPT
  files from disk but internally converts them to a Web Template representation before
  any FLAT path derivation — it never maps OPT archetype node IDs to FLAT paths
  directly.
- The `flatehr` Python tool (crs4): uses `web_template.json` as its template source,
  not the OPT.
- The openEHR REST 1.1.0 spec (in development, per Sebastian Iancu on Discourse,
  February 2026): formally specifies the `/example?format=FLAT` endpoint as the
  canonical way to discover correct FLAT path structure for a given template.

## Decision

**OPT-to-Web-Template transformation is a CDR responsibility. oehrpy treats the Web
Template endpoint as the authoritative output of that transformation and never
reimplements the transformation itself.**

Web Template generation is not an SDK concern. It is a **CDR responsibility**. The
moment an OPT is uploaded to any openEHR-compliant instance — EHRBase, Better Platform,
Nedap, eventually NeoEHR — the CDR becomes the authoritative source for what that
template's FLAT paths look like. The CDR has already performed the OPT → Web Template
transformation internally to handle composition validation and storage. Fetching the
Web Template via `GET /definition/template/adl1.4/{id}` with
`Accept: application/openehr.wt+json` simply accesses that already-computed view.

This is **spec-backed, not implementation-specific.** The openEHR REST specification
defines the Web Template endpoint as a standard interface. Better Platform had it first,
EHRBase adopted the same format, and it is being formalised in REST 1.1.0. Fetching the
Web Template from the CDR is portable across vendors.

This also means **the CDR version matters, not the SDK version.** If a CDR changes its
normalisation rules between releases (as EHRBase demonstrably did between 2.0 and
2.26.0), the Web Template endpoint automatically returns the updated view. An SDK that
hardcodes paths derived from OPT parsing would silently break; an SDK that fetches from
the CDR picks up the change on the next call.

In practical terms: the OPT is the *input* to the CDR; the Web Template is the *output*
that application developers should consume. oehrpy never sits between these two — it
consumes the output.

Concretely:

1. **`FlatValidator`** fetches the Web Template from the CDR via
   `GET /rest/openehr/v1/definition/template/adl1.4/{template_id}` with
   `Accept: application/openehr.wt+json`, then traverses the `tree` to build the set
   of valid FLAT paths. It does not parse OPT XML for this purpose.

2. **Composition builders** (`VitalSignsBuilder`, `ImmunizationBuilder`, and any
   future builders) source all FLAT path prefixes from the Web Template. Hardcoded path
   strings in builders must be verified against the Web Template, not the OPT.

3. **`BuilderGenerator`** (OPT → Python class code generation) is **scoped to metadata
   extraction only** — template ID, concept name, RM type, list of archetypes. It must
   not generate FLAT path strings, as these cannot be reliably derived from OPT XML.
   The code generator may be updated in a future version to accept a Web Template JSON
   as input instead.

4. **Web Template caching** is introduced to avoid repeated CDR round-trips. The
   `EHRBaseClient` caches fetched Web Templates in memory keyed by `template_id` for
   the lifetime of the client instance. A future version may support disk-based caching
   for reuse across sessions.

5. **Offline / Pyodide scenarios** (e.g., `FlatValidator` running in the browser via
   WebAssembly): a pre-fetched Web Template JSON must be bundled or supplied by the
   caller. OPT XML is not an acceptable substitute — the OPT is the *input* to the
   CDR, not the output that application code should consume. The Web Template can be
   exported from the openEHR Archetype Designer (Better) or fetched once from any CDR
   and cached for offline use.

6. **OPT parsing (`OPTParser`)** is retained in the public API for legitimate offline
   use cases: extracting template metadata, listing archetypes, and as an input to
   tooling workflows (e.g., the openEHR Explorer desktop app's template upload flow).
   Its scope is explicitly **not** FLAT path derivation.

## Alternatives Considered

### Alternative A: Derive FLAT paths from OPT using a normalisation algorithm

Implement the CDR's internal OPT → Web Template normalisation rules in Python (e.g.,
convert archetype concept names to snake_case, handle duplicate `id` suffixes).

**Rejected.** OPT-to-Web-Template transformation is a CDR responsibility, not an SDK
concern. The normalisation rules are not fully specified in any openEHR standard
document. They differ between CDR implementations (EHRBase vs. Better Platform) and
between CDR versions. Reimplementing them in Python would mean the SDK silently couples
to CDR internals — and if the CDR changes its rules (as EHRBase did between 2.0 and
2.26.0), the SDK produces wrong paths with no error. The integration testing experience
confirmed this: every attempt to derive FLAT paths from OPT analysis produced subtly
incorrect results that EHRBase rejected.

### Alternative B: Use the `/example?format=FLAT` endpoint to infer paths

Fetch a sample FLAT composition from the CDR's example endpoint and extract path keys
from it to build the valid-path set.

**Not adopted as primary approach, but complementary.** The example endpoint is
reliable for discovering correct paths, but returns a complete composition payload
rather than a structured Web Template tree. Building constraint metadata (required
fields, value types, allowed codes) from the example endpoint output would require
significant heuristic logic. The Web Template tree (`tree.children[]` with `rmType`,
`min`, `max`, `inputs`) is a better structured source. The example endpoint remains
useful as a developer tool and is documented in `docs/flat-format-learnings.md`.

### Alternative C: Use canonical JSON format exclusively, drop FLAT support

Abandon FLAT format and require callers to work only with canonical JSON (structured
COMPOSITION/OBSERVATION/etc. objects).

**Rejected for the SDK's primary use case.** Canonical JSON requires constructing the
full RM object graph, which is significantly more verbose and harder to use for
application developers. The FLAT format is the pragmatic choice for form-driven data
entry and integration workflows. It is also the format that EHRBase's simSDT endpoint
and the openEHR SDT spec are optimised for.

## Consequences

### Positive

- **Correctness**: FLAT paths are guaranteed to match what EHRBase expects, because
  they come from the same source EHRBase uses internally.
- **CDR compatibility**: If a path changes due to a template update or CDR upgrade, the
  SDK detects it at fetch time rather than silently producing invalid paths.
- **Spec alignment**: The approach matches the openEHR SDT specification and the
  direction of the upcoming REST 1.1.0 standard.
- **Clearer API contract**: `OPTParser` has a well-defined, honest scope. Callers are
  not misled into thinking OPT-derived paths are FLAT-safe.

### Negative

- **Live CDR dependency for path-sensitive features**: `FlatValidator` and composition
  builders that source paths dynamically require a reachable CDR (or a pre-fetched Web
  Template JSON). This adds a network dependency compared to the fully-offline OPT
  parsing approach.
- **Offline bundle maintenance**: Pyodide and other offline contexts require bundling
  Web Template JSON files alongside OPT files. This is two artefacts to keep
  synchronised per template.
- **`BuilderGenerator` scope reduction**: The current code generator produces FLAT
  path strings from OPT analysis. These must be removed or clearly marked as
  illustrative/unreliable, which is a breaking change to its generated output.

### Neutral

- The `OPTParser` class and `parse_opt()` function remain in the public API unchanged.
  Only the *claim* that they support FLAT path derivation is removed from documentation.
- Web Template caching is a net positive for performance but introduces statefulness
  into the `EHRBaseClient`.
